# Подключение оператора к диалогу пациент — INMYHEART

Документ описывает варианты, как оператор (администратор регистратуры) подключается к общению пользователя с Telegram-ботом при эскалации.

**Текущая реализация:** **Forum Topics** — при эскалации создаётся ветка «Клиент @username · id123» в группе `TELEGRAM_OPERATOR_CHAT_ID`; последующие сообщения клиента пересылаются в эту ветку.

**Docker:** бот работает в контейнере `bot`; переменные Telegram — из `.env` (см. [`DOCKER.md`](DOCKER.md)). Связь `user_id → topic` хранится в volume `/app/data/telegram_client_topics.json`.

---

## Вариант 1. Уведомление в Telegram-группу операторов (MVP, реализовано)

**Как работает**

1. Бот отвечает пациенту и сообщает, что переводит на оператора.
2. В группу `TELEGRAM_OPERATOR_CHAT_ID` уходит карточка: user id, маршрут, вопрос, AI-резюме.
3. Оператор **связывается с пациентом вне бота**: телефон регистратуры, личный кабинет, очный визит.

**Плюсы:** быстро внедрить, без доработки UI, соответствует ТЗ (эскалация без хранения ПДн в боте).  
**Минусы:** нет единого чата «пациент ↔ оператор» в Telegram.

**Настройка**

```env
TELEGRAM_BOT_TOKEN=...
TELEGRAM_OPERATOR_CHAT_ID=-1001234567890
TELEGRAM_HANDOFF_MODE=notify
```

Группа: создать супергруппу, добавить бота, получить chat id через `@userinfobot` или `getUpdates`.

**Запуск в Docker:**

```bash
docker compose up -d bot
docker compose logs -f bot
```

## Вариант 2. Прокси-чат (relay) — ответ оператора через бота

**Как работает**

1. При эскалации бот переводит сессию в режим `operator_active`.
2. Сообщения пациента пересылаются в группу операторов с меткой `#user123456`.
3. Оператор **отвечает reply** на сообщение в группе — бот доставляет текст пациенту от имени «Оператор INMYHEART».

**Плюсы:** пациент остаётся в том же чате с ботом; не нужен отдельный аккаунт оператора у каждого пациента.  
**Минусы:** нужна state-машина, авторизация операторов, таймауты сессии, модерация.

**Когда выбирать:** небольшая регистратура (2–5 человек), основной канал — Telegram.

**Реализация:** `TELEGRAM_HANDOFF_MODE=relay` (задел в коде; полная логика — следующий этап).

## Вариант 3. Forum Topics — отдельная «ветка» на каждого пациента (реализовано)

**Как работает**

1. Супергруппа операторов с **Topics** (форум), напр. [inmyheartclinic](https://t.me/inmyheartclinic).
2. При первой **эскалации** бот вызывает `createForumTopic` → «Клиент @username · id123456».
3. Карточка эскалации и дальнейшие сообщения клиента идут **в эту ветку**.
4. Связь `user_id → message_thread_id` хранится в `data/telegram_client_topics.json` (в Docker — volume `inmyheart-data`).

**Права бота:** администратор + **Управление темами** (Manage Topics).

**Настройка `.env`:**

```env
TELEGRAM_OPERATOR_CHAT_ID=-1003942732457
TELEGRAM_USE_FORUM_TOPICS=true
TELEGRAM_FORWARD_CLIENT_MESSAGES=true
TELEGRAM_NEW_TOPIC_AFTER_RESET=false
```

**Плюсы:** операторы не путают диалоги; история по клиенту в одной ветке.  
**Минусы:** без relay оператор не отвечает пациенту из ветки (relay реализован — см. `bot/relay.py`).

**Docker:** контейнер `bot`, `.env` как ниже. Не запускайте второй экземпляр бота локально — будет `Conflict` при polling.

## Вариант 4. Веб-панель оператора (FastAPI + WebSocket)

**Как работает**

1. Бот создаёт **тикет** в SQLite/PostgreSQL.
2. Оператор открывает `/admin/tickets` в браузере, видит очередь и live-чат.
3. Ответы из панели уходят пациенту через Bot API.

**Плюсы:** полный контроль, аудит, SLA, несколько каналов (Telegram + web).  
**Минусы:** больше разработки (backend, auth, UI).

**Когда выбирать:** дипломный этап 2+, интеграция с CRM, отчётность для руководства.

---

## Вариант 5. Внешний helpdesk (Usedesk, Jivo, Bitrix24)

**Как работает**

1. Webhook при эскалации создаёт тикет во внешней системе.
2. Оператор работает в привычном интерфейсе helpdesk.
3. Ответы (если поддерживается API) возвращаются в Telegram через бота.

**Плюсы:** готовая очередь, CRM, статистика.  
**Минусы:** платные тарифы, интеграция, зависимость от вендора.

**Когда выбирать:** клиника уже использует helpdesk.

---

## Вариант 6. Только офлайн-канал (телефон / регистратура)

**Как работает**

1. Бот сообщает: «Перевожу на оператора» + телефон +7 (495) 123-45-67.
2. Уведомление оператору опционально (вариант 1).
3. Оператор **перезванивает** или ждёт звонка пациента.

**Плюсы:** минимум разработки, привычно для медицины.  
**Минусы:** пациент уходит из чата; нет сквозной истории в мессенджере.

**Когда выбирать:** MVP диплома, если relay не успеваете реализовать.
